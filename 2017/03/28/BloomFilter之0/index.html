BloomFilter 初始零

Veröffentlicht am 2017-03-28   |   更新于 2017-03-28

&esmp;&esmp;爬虫爬取过程中，对于url的去重是较为重要的。爬虫脚本的每一次运行，都需要判断是否这个url以及这个url
中的数据曾被获取过。所以研究了一下关于url去重，大致的想法是使用BloomFilter。由于每一个url的地址长度较长，将所有url的信息
全部读入到内存中效率太低，BloomFilter的原理是 找到K个适合的hash函数，每个高效的hash函数可以将url映射到内存的一位上，为了避免
hash冲突，所以使用多个hash函数进行运算。每个hash函数加密得到一个位数值，数值可以在一个bit数组进行映射，在bit数组中
找到该数值对应的位置，并把该位置的值设置为1（默认为零）。K个hash函数就将K位设置为零。需要判断url是否被爬取过，则只需将url
使用K个hash函数加密，得到K个值，判断K个值在Bit数组中的映射位置是否全部为1，全部为1的话，那么为了避免误判，可以继续去硬盘中的数据库
去找，或者直接认定该url存在。但只要映射的位置，有不为1的情况，那么该数据一定不存在。
&esmp;&esmp;原理很简单，具体的实现有几大困难。1.确定hash函数 2.确定bit集合大小 其实想想也不难，毕竟都是有具体的公式的

贴一个更加优化的：
上面说到的BloomFilter还紧紧是单机内存的，在淘宝这个环境下，显然是不适用的。那么我们如何把他变成分布式的呢？看了标题我想你已经知道了，对了，那就是redis。
BloomFilter需要的bit数组与redis的bit操作真是完美契合啊。利用redis的高性能以及通过pipeline将多条bit操作命令批量提交，实现了多机BloomFilter的bit数据共享。唯一需要注意的是redis的bitmap只支持2^32大小，对应到内存也就是512MB,数组的下标最大只能是2^32-1。不过这个限制我们可以通过构建多个redis的bitmap通过hash取模的方式分散一下即可。同时利用上面的公式计算一下：万分之一的误判率，512MB可以放下2亿左右的数据，而目前全网的uv也就8千万，所以，你懂的。链接戳我
